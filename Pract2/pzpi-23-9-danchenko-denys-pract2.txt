3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
Назва системи: Spotify
Розробник: Spotify Technology S.A.
Призначення системи:
Spotify – це потоковий музичний сервіс, призначений для онлайн-прослуховування аудіоконтенту, зокрема музики, подкастів та інших медіа. Система використовується широким колом користувачів, включаючи індивідуальних слухачів, виконавців, авторів подкастів та бізнес-користувачів.
Основними функціями системи є:
	відтворення музики в режимі онлайн;
	створення та управління плейлистами;
	персоналізовані рекомендації на основі вподобань користувача;
	пошук музики, виконавців та альбомів;
	офлайн-доступ до контенту (для підписників);
	синхронізація між різними пристроями.
3.2 Архітектурний стиль системи
Архітектура Spotify є мікросервісною, клієнт-серверною та розподіленою. Основним архітектурним стилем можна вважати мікросервісну архітектуру, оскільки система складається з багатьох окремих сервісів, які відповідають за конкретні функції: авторизацію користувачів, пошук музики, рекомендації, плейлисти, оплату підписки, обробку аудіопотоків та зберігання даних.
Також Spotify використовує клієнт-серверну архітектуру, бо користувач взаємодіє з системою через мобільний застосунок, вебверсію або десктопну програму, а основна обробка запитів виконується на серверній стороні.
Вибір такої архітектури обґрунтований тим, що Spotify має велику кількість користувачів і повинен швидко обробляти запити, забезпечувати безперервне відтворення музики, персоналізовані рекомендації та стабільну роботу сервісу. Мікросервісний підхід дає змогу окремо масштабувати різні частини системи, оновлювати сервіси незалежно один від одного та швидше усувати помилки без повної зупинки платформи. Spotify Engineering прямо описує роботу з мікросервісами у своїх матеріалах.
3.3 Основні компоненти системи
Архітектура Spotify об’єднує набір компонентів, які можна умовно поділити на кілька основних груп залежно від їхнього призначення.
До першої групи належить інтерфейс користувача, представлений мобільними застосунками, вебверсією та десктопним клієнтом. Взаємодія з сервером здійснюється через HTTP/HTTPS API, а передача аудіо реалізується за допомогою потокового стрімінгу. Клієнт забезпечує пошук контенту, відтворення музики та керування плейлистами.
Серверна частина побудована за мікросервісною архітектурою та розгорнута у хмарному середовищі. Окремі сервіси відповідають за автентифікацію, пошук, рекомендації та обробку запитів, взаємодіючи між собою через API та асинхронні повідомлення. Для управління сервісами використовуються контейнерні технології.
Рівень бізнес-логіки реалізує основні алгоритми платформи, зокрема персоналізовані рекомендації на основі аналізу даних користувача та керування підписками.
Рівень зберігання даних включає реляційні та NoSQL бази даних, а також розподілені сховища для зберігання аудіоконтенту та метаданих.
Допоміжні сервіси забезпечують стабільність роботи системи та включають кешування, CDN для доставки контенту, балансування навантаження, моніторинг і механізми безпеки.
3.4 Взаємодія компонентів
Взаємодія компонентів у системі реалізується за клієнт-серверною моделлю з використанням мікросервісної архітектури. (мобільний, веб та десктопний) взаємодіють із серверною частиною через API Gateway за протоколом HTTPS, передаючи запити на обробку та отримуючи відповіді у вигляді метаданих або посилань на аудіопотік.
API Gateway виступає єдиною точкою входу до системи та виконує маршрутизацію запитів до відповідних мікросервісів. Зокрема, запити на автентифікацію обробляються Auth Service, пошукові запити Catalog (Search) Service, а рекомендації формуються Recommendation Service на основі даних про користувача та історію прослуховування.
При відтворенні музики клієнт ініціює запит, який проходить перевірку автентифікації, після чого Streaming Service отримує інформацію про трек із бази даних метаданих та формує запит до CDN. Безпосередня передача аудіопотоку здійснюється через CDN, що дозволяє зменшити затримки та забезпечити стабільне відтворення контенту.
Для взаємодії між сервісами використовуються як синхронні запити (REST API), так і асинхронні механізми обміну повідомленнями. Синхронізація даних між компонентами забезпечується через API-запити, обробку подій та оновлення розподілених баз даних, що дозволяє підтримувати актуальний стан плейлистів, історії прослуховування та рекомендацій на різних пристроях користувача. Зберігання даних реалізовано за допомогою окремих баз даних для різних типів інформації, що забезпечує масштабованість і незалежність компонентів системи.
 
Рисунок 3.1 – Діаграма послідовності процесу відтворення аудіо в системі Spotify
Діаграма послідовності відображає процес взаємодії компонентів системи під час відтворення музичного треку. На ній показано послідовність запитів від користувача до клієнтського застосунку, API Gateway та мікросервісів, а також отримання аудіопотоку через CDN.
 
Рисунок 3.2 – Компонентна діаграма архітектури системи Spotify
Компонентна діаграма ілюструє основні компоненти системи Spotify та зв’язки між ними. Вона демонструє взаємодію клієнтської частини з API Gateway, мікросервісами та рівнем зберігання даних, а також використання CDN для доставки аудіоконтенту.
3.5 Масштабованість та надійність
Система Spotify спроєктована з урахуванням високих вимог до масштабованості та надійності, що обумовлено великою кількістю одночасних користувачів і значним обсягом оброблюваних даних.
Масштабованість системи забезпечується за рахунок мікросервісної архітектури, яка дозволяє незалежно масштабувати окремі компоненти залежно від навантаження. Для розподілу запитів між серверами використовуються механізми балансування навантаження, що забезпечують рівномірний розподіл трафіку та запобігають перевантаженню окремих вузлів.
Відмовостійкість досягається завдяки розподіленій архітектурі та дублюванню критичних сервісів. У разі відмови одного з компонентів система здатна автоматично перенаправляти запити на інші доступні вузли, що забезпечує безперервність роботи сервісу.
Для підвищення продуктивності застосовується кешування, зокрема з використанням систем на кшталт Redis, що дозволяє зменшити час доступу до часто використовуваних даних та знизити навантаження на бази даних.
Реплікація даних використовується для забезпечення їхньої доступності та надійності. Дані зберігаються у кількох копіях на різних серверах, що дозволяє уникнути втрати інформації у випадку збоїв.
Інфраструктура системи побудована на основі хмарних технологій із використанням контейнеризації. Це забезпечує гнучке розгортання сервісів, автоматичне масштабування та ефективне управління ресурсами, що є критично важливим для високонавантажених систем.
3.6 Приклади програмної реалізації
Для демонстрації архітектурних принципів наведено приклади взаємодії з API музичного сервісу. Вони показують клієнт-серверну взаємодію, використання REST API та обробку даних у форматі JSON.
Перший приклад демонструє отримання інформації про трек через Spotify Web API. Клієнт надсилає HTTP-запит до серверної частини, використовуючи токен авторизації:
1  using System.Net.Http;
2  using System.Net.Http.Headers;
3  
4  var client = new HttpClient();
5  string accessToken = "USER_ACCESS_TOKEN";
6  string trackId = "TRACK_ID";
7  
8  client.DefaultRequestHeaders.Authorization =
9      new AuthenticationHeaderValue("Bearer", accessToken);
10 
11 var response = await client.GetAsync(
12     $"https://api.spotify.com/v1/tracks/{trackId}");
13 
14 string result = await response.Content.ReadAsStringAsync();
15 Console.WriteLine(result);
Цей приклад демонструє принцип клієнт-серверної архітектури: клієнтський застосунок надсилає запит до API, а сервер повертає дані про трек у форматі JSON.
Другий приклад ілюструє умовну роботу сервісу рекомендацій. Після прослуховування треку інформація про дію користувача може передаватися до окремого сервісу, який аналізує історію прослуховування та формує персоналізовані рекомендації:

2  {
3      public string UserId { get; set; }
4      public string TrackId { get; set; }
5      public string Action { get; set; }
6      public DateTime CreatedAt { get; set; }

18 
19  await recommendationService.ProcessEventAsync(listeningEvent);
20 }
Цей приклад демонструє подієво-орієнтований підхід, коли дія користувача обробляється окремим сервісом без прямої залежності від інших компонентів системи.
Наведені приклади показують основні архітектурні принципи Spotify: використання API для обміну даними, розділення функціональності між сервісами, обробку подій користувача та формування персоналізованого контенту.
3.7 Переваги архітектурних рішень Spotify
Архітектура платформи Spotify забезпечує низку суттєвих переваг порівняно з традиційними підходами до побудови програмних систем:
1.	Гнучкість і масштабованість – використання мікросервісної архітектури дозволяє незалежно масштабувати окремі сервіси залежно від навантаження, що забезпечує ефективну обробку великої кількості користувачів;
2.	Висока доступність – розподілена інфраструктура та використання CDN дозволяють забезпечити безперервний доступ до контенту та стабільне відтворення музики;
3.	Продуктивність – застосування кешування та оптимізованих механізмів доставки контенту значно зменшує затримки при отриманні даних;
4.	Персоналізація – використання алгоритмів аналізу даних і машинного навчання дозволяє формувати індивідуальні рекомендації для кожного користувача;
5.	Відмовостійкість – ізоляція сервісів та дублювання даних дозволяють системі продовжувати роботу навіть у разі відмови окремих компонентів;
6.	Глобальна доступність – використання хмарної інфраструктури та розподілених серверів забезпечує швидкий доступ до сервісу для користувачів у різних регіонах світу.
